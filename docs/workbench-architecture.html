2. Multi-Flow Architecture

Every LLM flow follows the same shape: fabricated state → assembled prompt → LLM call → structured output. The generic engine handles all flows. Project-specific hooks.py registers each flow with its trigger function and assembler. 
FLOW REGISTRY (hooks.py — project-specific)

  FLOWS = {
  ┌─────────────────┬───────────────────────────┬──────────────────────────┬───────────────┐
  │ Flow ID         │ Trigger Function          │ Assembler(s)             │ Output Schema │
  ├─────────────────┼───────────────────────────┼──────────────────────────┼───────────────┤
  │ trigger         │ run_trigger()             │ assemble_evaluator_prompt│ LogEvaluation │
  │                 │                           │ assemble_drafter_prompt  │ CreateDraft   │
  ├─────────────────┼───────────────────────────┼──────────────────────────┼───────────────┤

  Each flow is just a different pair of functions. Adding a new flow = one FlowConfig entry.

3. Trigger Pipeline: Interval Gating, Routing & Draft Groups

With commit_analysis_interval = 3:

  Commit 1 → run_trigger() → counter 0→1 → deferred_eval (no LLM)  [0.2s]
  Commit 2 → run_trigger() → counter 1→2 → deferred_eval (no LLM)  [0.2s]
  Commit 3 → run_trigger() → counter 2→3 ≥ threshold
      │
      ├── reset counter to 0
      ├── gather deferred [commit1_dec, commit2_dec]
      ├── combine diffs: "--- Commit h1: ... ---\n{diff1}\n--- Commit h2: ..."
      │
      ├── Stage 1: Analyzer on combined CommitInfo                [~30-120s]
      │     → classification, tags, summary, topic_suggestions
      │     → (trivial? → skip all, fast return)
      │
      ├── Stage 2: Evaluator on combined CommitInfo               [~30-120s]
      │     → per-STRATEGY decisions: { "building-public": draft, "product-news": hold }
      │     → each with: action, reason, angle, arc_id, topic_id, media_tool, ...
      │     → queue_actions: { strategy → [supersede/merge/drop] }
      │
      ├── Routing (deterministic, no LLM)                         [instant]
      │     route_to_targets(strategy_decisions, config.targets)
      │
      │     Strategy decisions          Target config             Routed actions
      │     ┌──────────────────┐       ┌──────────────────┐       ┌──────────────────┐
      │     │ building-public:  │       │ x-dev:           │       │ x-dev:     draft │
      │     │   action: draft   │──────→│   strategy: bp   │──────→│ x-community:     │
      │     │                   │       │   primary: true   │       │   draft (source  │
      │     │ product-news:     │       │ x-community:     │       │   x-dev fired)   │
      │     │   action: hold    │──┐    │   strategy: bp   │       │ li-product: skip │
      │     └──────────────────┘  │    │   source: x-dev  │       │   (strategy=hold) │
      │                            │    │ li-product:      │       │ li-community:     │
      │                            └───→│   strategy: pn   │──────→│   skip (source   │
      │                                 │   primary: true   │       │   li-prod skipped)│
      │                                 │ li-community:    │       └──────────────────┘
      │                                 │   strategy: pn   │
      │                                 │   source: li-prod│       Draft groups:
      │                                 └──────────────────┘       { bp: [x-dev,
      │                                                               x-community] }
      │
      ├── Stage 3: Drafter (one LLM call per draft group)         [~30-120s]
      │     group "building-public":
      │       → 1 drafter call → content + variants[x-dev, x-community]
      │       → Draft inserts with target_id, is_intro, preview_mode
      │
      └── return 0

  ReplayEngine just calls run_trigger() for each commit.
  Gating, batching, routing, draft groups all happen organically inside.

5. Interaction Flows (Time-Based Cost Model)

LAYER 0: Prompt Diff + Routing Sim (instant, $0)
Primary exploration tool. Use for 90% of work. Works for ALL flows.

  User selects flow ──→ User adjusts state ──→ User selects prompt version
          │                       │                         │
          └───────────────────────┴─────────────────────────┘
                                    │
                    StateFabricator + PromptCapture
                    calls the flow's assembler_fn with fabricated state
                                    │
                                    ▼
                    PromptSnapshot { sections[], tokens, omitted[] }

  Trigger flow bonus: Routing Simulation (also instant)
                    RoutingCapture
                    calls route_to_targets() with fabricated/mock strategy decisions
                                    │
                                    ▼
                    Target actions, draft groups, skipped targets (and why),
                    scheduling constraint hits — all without LLM cost

  Expert flow example:
    State: snapshot "mid-lifecycle" + draft with OAuth content
    Input: user_message = "make it more technical, show the code"
    Result: Expert assembled prompt showing:
      ## Project Context, ## Author Identity, ## Project Summary,
      ## Current Draft (content + media), ## User Message, ## Escalation
    → See exactly what the expert LLM would receive. Zero cost.


LAYER 1: Single Execution (~1-5 min, $0)
One flow execution with full capture. Works for ALL flows.

  PipelineRunner.run(flow, state, input)
          │
          ▼
  Calls the flow's trigger_fn, captures prompts + output + mutations
          │
          ▼
  CapturedResult — prompts, routing (if trigger), output, db_mutations

  Time varies by flow:
    trigger:     ~2-5 min (analyzer + evaluator + drafter)
    gatekeeper:  ~30-60s  (single Haiku call)
    expert:      ~1-2 min (single drafter-model call)
    media_spec:  ~1-2 min (single drafter-model call)
    discovery:   ~3-8 min (2 LLM passes + brief gen)


LAYER 2: Commit Sequence (~5-15 min, $0, trigger flow only)
Sequential run_trigger() calls. Gating and batching happen organically.

  ReplayEngine.run(commits[:9], prompt_versions={...})
          │
          ├── commit 1 → run_trigger → deferred (0.2s)
          ├── commit 2 → run_trigger → deferred (0.2s)
          ├── commit 3 → run_trigger → BATCH [1,2,3] (~3 min)   ──┐
          ├── commit 4 → deferred                                   │ progress
          ├── commit 5 → deferred                                   │ via SSE
          ├── commit 6 → BATCH [4,5,6] (~3 min)                     │
          ├── commit 7 → deferred                                   │
          ├── commit 8 → deferred                                   │
          └── commit 9 → BATCH [7,8,9] (~3 min)                   ──┘
          │
          ▼
  ReplayResult { per_commit[], batch_points[], final_snapshot }


LAYER 3: Parallel Prompt Comparison (~10-20 min, $0)

9. Maintenance & Drift Detection

TWO TYPES OF COUPLING:

  Type 1: Tight (fails loudly — good)
  hooks.py calls run_trigger(), assemble_*_prompt(), Expert.handle() directly.
  Signature changes → TypeError immediately.

  Type 2: Loose (drifts silently — needs detection)
  WorkbenchState has fields for arcs, topics, strategies, etc.
  New entity added → nothing breaks, but can't fabricate the new thing.
  New LLM flow added → workbench doesn't know it exists.


DRIFT DETECTION (CI tests in drift.py):

  test_workbench_covers_domain_entities()
    Compares domain models to WorkbenchState fields.
    Fails with: "Domain entities not in WorkbenchState: {campaigns}"

  test_workbench_covers_all_flows()
    Compares known LLM flows to FLOWS registry.
    Fails with: "Flows not registered in workbench: {campaign_eval}"

  test_assembler_signatures_match()
    Inspects assemble_*_prompt() signatures vs. hook wrappers.
    Fails when new required parameter added.


WHAT STAYS IN SYNC AUTOMATICALLY (no maintenance):

  ✓ Captured LLM output — raw Pydantic model, new fields appear for free
  ✓ Prompt sections — parsed dynamically on "---\n## " delimiters
  ✓ Snapshots — full DB copy, includes any new tables/data
  ✓ Routing behavior — calls route_to_targets() directly
  ✓ Schema changes — CapturedResult stores JSON, not typed fields


WHAT NEEDS MANUAL UPDATE (~15-30 min per change):

  ✗ New domain entity → add to state.py + overrides.py + drift test
  ✗ New LLM flow → add FlowConfig to hooks.py + input model to state.py
  ✗ New required param on assembler → update hook wrapper


CONVENTION (CODING_PRACTICES.md):

  "When you add a new domain entity or LLM flow:
   update workbench/state.py, overrides.py, hooks.py.
   Same rule as REUSABILITY.md — update when interfaces change."

  • Pydantic models as single source of truth (validation + UI generation + DSL)
